什么是微服务-译文

微服务：如何更快交付软件？

随着软件组织寻找一种更快的方式去交付软件，专业术语微服务得到了很多持续关注。传统上来讲，开发团队已经构造了一个运行他们业务的单体架构，并且这些单体架构任然可以为一些依赖较少的应用工作。所以为什么这么急冲冲的去采用微服务呐？因为微服务是一个能改善软件组织速度，独立性，和创新性的引擎。

什么是微服务？

微服务是指一种不同团队用于同一目的的服务的一种架构方法，松耦合（如果每个服务必须去协同更新，他就不是松耦合）和有界上下文（如果你必须知道周围的服务，这就不是有界上下文）是这种面向服务架构的关键属性。换句话说，微服务是独立的，更小的代码片段，是一个系统的一部分但是不需要去更新和知道周围的服务。

当开发者们选择微服务和微服务设计模式去创造一个新的应用或者重构一个旧应用，这样会帮助软件跨团队和更好的进程更快的发布。微服务方法和工具让团队适应不断变化的需求，加速交付新特性给客户。

微服务模式与传统单体架构相反，传统架构由很紧密的模块组成，很少发布，不得不作为一个单元去扩展。虽然传统架构对于某些团队和某些应用还可以工作的很好，但是微服务在一些需要更敏捷和更强的拓展能力的公司很流行。

微服务的特点和操作是不同的：

  • 小型业务的集合，每个服务实现一个业务
  • 在自己的进程运行，通过HTTP API 交流
  • 能够独立于其他应用中的服务进行发布。升级，拓展和重启
  • 当被自动化系统管理时，微服务可以频繁在实时应用中频繁更新并且不会对用户造成不好的影响。

今天，各行各业–从零售到金融服务到制造业，都在使用微服务构建新的应用，分解单体架构。然而不是所有的应用都应该用微服务，因为微服务不只是单纯的代码重写，他们需要不同的思维方式，方法和操作模式。

为什么用微服务？

技术团队可以更积极的相应客户的需求。当团队采用微服务的时候可以在客户有需求的时候快速交付，不再受限于固定的发布计划。
更大的软件产出量。微服务在Agile和DevOps原则上建立，帮助团队并行工作的时候快速在离散的服务上迭代。
组织改进系统的可扩展性和可靠性。一个成功的微服务就是保持发展，他严重依赖可重复的自动化，支持细粒度的服务，使用模式设计去保持系统甚至能在单个组件故障的时候运行。

微服务 VS 传统架构

微服务

  • 聚焦一点

只做一件事， 做得很好！微服务的目标是在一个具体的的问题领域，并且包含该领域的每件事（包括数据）去拥有自己的体验，微服务的“微”是范围不是大小。

  • 是松耦合

一个为微服务架构要求服务能够自给自足，并且避免硬编码引用其他服务。

  • 是持续交付的

微服务是一个为了团队能够满足团队能够应付持续变化的需求的想法。能够尽快交付到市场，微服务可以通过自动化定期交付到产品。

  • 有独立的团队拥有服务的生命周期。

微服务的转型不仅是和团队结构有关也是和技术有关。微服务被独立的团队构造，发布，运行。不是每个服务都需要这样，但是这是一种强大的模式，对于商业型的关键服务来说。

  • 有设计模式和技术在扩展上强调分布式。

一个微服务依赖一个服务发现能力，信息，网络路由，故障检测，日志，存储，身份等等的集合。团队不能用产生一个单体架构软件相同的方法和工具去构建和运行微服务。

传统架构（单体）

  • 聚焦比较广泛

用紧密联系的软件包试图一次性解决许多业务的挑战，创造了许多代码依赖。

  • 紧耦合

系统通常是一个不谨慎按照步骤来就无法部署的相互依赖的组件缠绕的网。

  • 依赖按计划交付

应用按照计划时间被开发和更新。通常表现为按季度或者年份。

  • 有许多拥有生命周期的团队

项目团队负责构建第一代软件的迭代，然后被分开去做下一个任务。软件被交付给另一个团队去维持。

  • 有设计模式和技术，流程优先

孤立的工具和流程，专注于关键的开发阶段，QA，和发布到生产环境，产生单体软件。

当你准备去构建微服务架构的时候，要留意什么？

微服务不是任何组织或者应用都适合的。如果你不正确的应用，一个微服务架构能造成更高的耗费。如果你考虑开始作微服务，考虑一下下面的几个关键问题会有一些帮助。

  • 你的组织准备好了吗？

向微服务的转型，既是组织转型，也是技术转型。团队必须要准备接受一个自动化中心，可持续交付的方法来构建软件。你的公司准备好去移除功能上的独立和有自给自足的团队去构建和操作服务吗？你能够改变管理的进程容忍一个无人参与的部署流程吗？你考虑这些问题的方式会让你知道你准备的怎么样了。

  • 你有过于着急的开发者吗？

不是任何应用都适用微服务。匆忙去“微服务所有”，开发者大概会花费大量时间在业务不需要去改变的已有应用的编码上。使用量低的应用或者不对应服务关键性原则最好离开微服务，待在单体状态。微服务提高了敏捷性，然而带来了复杂性，确保你需要敏捷性。

  • 你的服务协调吗？

微服务之间是松耦合的，是不断变化的。你怎么找到当前服务的URL，或者流量路由到一个弹性数量的服务实例？服务之间怎么交换数据？在许多情况里面，你今天现有的技术去应对服务发现，负载均衡，消息传递对于微服务的动态变化是远远不够的。你有奉献精神和预算去投资在改变上吗？

  • 对于复杂的环境，你未来的管理是怎么样的？

随着应用和去管理的服务的数量增长，操作的风险也增长了。没有新方法去解决，在数百或者数千台服务器上创建微服务绝对会让管理人员头大。在线下的机器上打补丁或者升级是不是很困难？你能够追踪依赖然后确认在危险中的应用吗？让大量的微服务实例和最新的应用配置同步有多么困难你知道吗？你创建微服务平台的组件和这些组件和服务在哪里运行这些都会在未来的时间里对你组织的敏捷性造成重大的影响。

鉴定微服务的引导准则

你的团队写的代码比以前更多，这是微服务吗？下面是一些关于应用和系统最好去转化成微服务的

  • 成倍数的变化率

如果你的系统的一部分需要往与该系统速度和方向不一样的地方发展，就应该分开这部分转化为微服务，能让这个部分有一个独立的生命周期。（例如将电子商务系统的购物车功能从搜索或者推荐引擎中分离出来）

  • 独立的生命周期

当代码提交到一个模块的生产流程需要独立的生命周期，就把这个转化成具有自己代码库和CI/CD 管道的微服务。

  • 独立的扩展能力

当装载或者吞吐是系统独一无二的一部分，这部分也可能是拓展要求，这都让这个应用做一个微服务是一个理想的选择–以一个不同的速度拓展。

  • 孤立故障的需求

当不能选择一个故障的应用的时候，创建一个有故障转移选项的微服务去隔离不可靠的来自系统其他部分的依赖。

  • 需要去简化永久依赖的交互

当你需要在频繁变化的永久依赖中保护系统的时候，创建一个微服务（例如支付）

  • 选择的自由

通过创建微服务，允许团队去依赖他们自己优先的技术栈。

从单体到微服务的三步

抓着传统档案的团队需要去做出现代化决定。许多团队选择拆分他们的单体架构，转化为微服务，他们的最终目标是转移到原生的云应用。可用的耦合工具，最好的实践过程和技术可以让转变更加容易。

  • 步骤一：开始单体架构

我们认为单体架构是一个合适的初级阶段选择，但是当太多开发者和其他人员开始使用相同的代码库的时候，问题出现了。开发者认为产品组合很复杂，文档有限，资源太紧耦合了，他们不能学习新技能，因为他们要维护传统应用和基础设施，这就是你转去微服务的时候。

  • 步骤二：找到缝合线去发现上下文

从一个单体上去查找微服务的关键是去找到应用的“缝合口”–所以你就能知道有界上下文。然后将有界上下文取出。但是首先，你要给有界上下文放一个栅栏看看是谁在调用，然后就可以知道谁在调出，谁在调入
，依赖关系是什么，约束是什么？这会让你了解耦合，还有与之关联的事件。接着，然后你可以放一个API栅栏在有界上下文外面，然后开始拉出有界上下文到他自己的应用中。

  • 步骤三：科学的方法重构代码：Mikado 方法

对于每个改变，找到依赖关系里面出现错误的那个地方，修复他们，然后继续修复。这是一个被证明过的处理重构的深度优先的方法。

-Thinking Architecturally

架构师思想—第一次翻译

前言

Rich Hickey 曾经说过编程者要知道每件事的利益，不用权衡其它(tradeoffs of nothing)。。。这个一个能让项目去打击开发者和惹恼客户的路的方法。但是作为一个架构师，你必须要考虑每个新的库，开发语言，设计模式，方法，然后根据你拿到的不完整的信息，快速做出决定。你要怎么考虑项目中不可避免的技术选择？你怎么平衡矛盾的会议日程，你怎么保持你的团队在没有使用最新的技术的时候保持激情。

作为一个架构师，这是你的责任去在团队的技术行程中去有效的引导他们，接下来，你要跟着一个从发现新技术到保持一个新技术在你的团队中的一个弧线。你将学会权衡的重要，你怎么分析新技术，你怎么高效的做出必要的框架选择。你还可以了解到你做的正确的功能选择能够映射到你的codebase。
在你继续阅读的时候，我想做一些关于标题的解释(这里我不太清楚作者是想说称号头衔还是架构这个词)。在软件行业里面，架构没有一个好的定义，询问20个开发者给架构一个定义，你会得到20个不同的回答。还有很多组织自大的说他们没有架构师，此外，“架构”这个标题可能被严密的警戒起来在某些组织里面（1），很多人做了很细微的区分，我希望没有架构师头衔的可以看看这本书，尽可能发现一些有用的东西，严格来讲，这本书涉及到了技术领导，高级开发者，初级开发者，当然还有架构师，无论你真正的头衔是什么。

1.我是一个“高级开发者”做了很多年的架构工作，然后我最终被授予了架构师头衔
无论你的头衔是什么，你都可能通过看书成为一个架构师。你需要机会去练习架构的过程。这是为什么每个章节都有一套Katas【注：Kata的意思是一种结构化的方式去做事情或者练习–我觉得可以理解为和力扣差不多的网站】或者练习，去帮助你掌握概念。Katas 起源于武术的训练，但是这个概念现在不止用于武术道场。在Fred Books 中有一句比较著名的话：我们怎么去成为伟大的设计师？伟大的设计师设计（How do we get great designers? Great
designers design, of course.）这就意味着这也是成为架构师的方式，架构是一个巨大的系统。
然而，一般的架构师也只是在一小撮应用程序上，没有必要的联系你怎么可能成为大师。专业运动员花费在练习的时间远远大于他们在体育竞技里面的的时间(Profes‐
sional athletes spend far more time practicing their sport than they do playing in games that “count.”)；专业的音乐家和演员在现场表演之前会进行无数次的预演，但是在一般的等级之外，大多数架构师在工作中学习，意味着他们会在工程项目上产生错误，这就不用疑惑有这么多项目不能体现预期的价值。
建筑katas的概念产生于一场研讨会，Neward, Neal Ford, Mark Richards, Matt Stine, Simon Brown, 作者和一些其他人在前几年主导了一个以让参与者有一个能够解决模拟现实世界的问题和架构问题的机会。

技术的变化

如果你在软件世界呆过一段时间，你肯定会知道技术的变化很快，这是不可避免的。在我的早期职业生涯，我因为我认为我错过了技术的浪潮感到沮丧，那个时候我不喜欢技术像巴士一样，每十五分钟来一趟。
做为架构师，我们要记住我们不能停止改变，但是我们能掌管它。在接下来的章节我们会介绍过去可以教会你的，提示你不能预测未来的技术，为什么你不应该快速的丢掉传统技能。

在此前我们见过这个吗？

这些不能记住过去的最终都会重蹈覆辙
对于大多数开发者，从他们第一次学习开发语言的时候，最初的语言就成了我们的引路灯，我们通常会会根据我们最初的经历对比其他语言，然而我们最先的语言已经打下了深厚的基础，这个可能会让我们比较局限。许多开发者不怎么高兴有新特性添加到他们最爱的开发语言。当java 8 添加了lambda表达式，一些开发者不明白为什么他们的语言需要这些新特性。我相信有一部分人更钟意于过去的版本。

分布式

概念方面

微服务的概念

就是拒绝大型应用，把服务拆分，分成一个个的服务，各个服务相互独立。就是把单独的应用程序，开发成一套服务，每个小服务在自己的进程里面运行，使用轻量级机制通信，通常是HTTP API，这些服务都围绕业务能力构建。并通过完全自动化部署机制独立部署，可以使用不同的编程语言实现和不同的数据存储技术，保持最低限度的集中式管理。

集群&分布式&节点

集群：物理形态，一堆机器协同工作，使用几台服务器协同工作
分布式：一种工作方式
节点：集群中的一个服务器

远程调用

分布式系统中，各个小服务之间使用HTTP+JSON方式完成远程调用。数据用JSON封装起来传递

负载均衡

常见的算法

  1. 轮询：为第一个请求选择健康池的第一个后端服务器，然后按顺序往后依次查找，知道最后一个，然后循环。
  2. 最小连接：优先选择连接数最少，也就是压力最小的服务器，会话时间较长一般使用这种。
  3. 散列：哈希

    服务注册/发现&注册中心

    两个服务之间相互调用，不知道对方服务有几个正常运行，用服务注册

    配置中心

    每个服务都有自己的配置，为了不一更换机器就重新配置，所在服务在配置中心获取自己的配置

    服务熔断和降级

    为了防止一个服务的不可用造成雪崩效应，使用容错机制保护服务

服务熔断：请求某个服务超时的时候，开始保护机制，直接启用默认数据，不再访问该服务
服务降级：当系统处于高峰期，资源紧张，可以让非核心业务降级运行。降级：某些业务不做处理或者简单处理

    怎么检查：maven -v

src/test/java  测试代码

@controller标记在一个类上之后，这个类就是springMvc的控制类，然后搭配spring的话就要去配置，但是我用的是spring boot，不用配置，这个真的很方便，我标记这个类之后，只是定义了他。
@RequestMapping这玩意儿才是处理请求的，但是这个已经被代替了，过时了
  • @GetMapping*：处理get请求
  • @PostMapping*：处理post请求
  • @PutMapping*：处理put请求
  • @DeleteMapping*：处理delete请求
    @RequestParam：将请求参数绑定到你控制器的方法参数上（是springmvc中接收普通参数的注解）
语法：@RequestParam(value=”参数名”,required=”true/false”,defaultValue=””)
value：参数名
required：是否包含该参数，默认为true，表示该请求路径中必须包含该参数，如果不包含就报错。
defaultValue：默认参数值，如果设置了该值，required=true将失效，自动为false,如果没有传该参数，就使用默认值
  • model.addattribute()*：往前台传数据，可以传对象，可以传List，通过el表达式 ${}可以获取到
  • 更改端口号*：server.port=####

我的第一篇博客

个人博客的快速搭建

这是我的第一篇技术性博客，该说些什么好叻，不如就教大家怎么搭建自己的博客吧！
我用的是GitHub pages和Hexo ,Hexo是一个快速、简洁且高效的博客框架

使用GitHub page的原因

  1. 这是静态网页，虽然刚开始进去的时候比较慢，但是进入界面之后页面跳转就很快咯！
  2. 纯白嫖，咋也不花钱，不用买服务器，很自由的一个博客，还可以自己设计主题，当然也能用别人做好的，在Hexo的Themes里面有很多很好看的哦！
  3. 域名可以自己起，我的域名太正式了。。。
  4. 数据绝对安全，毕竟GitHub嘛，全球最大的同性交友网站哈哈哈！还能控制历史版本，也能恢复任意版本！

    环境准备

    GitHub准备

  • git，git操作命令不熟悉可以去B站补补课

  • 一个GitHub账号

  • 一个新建的GitHub仓库，名字要是dindindin.github.io dindindin是你自己起的名儿，爱起啥名儿起啥名儿。

    必要的软件

  • 安装node.js、npm，Hexo，安装的时候这个npm的时候如果出现问题，就本地全局都安装一遍，然后把npm的全局路径加到系统变量的path中。
    Windows用户建议使用程序安装node.js，$ npm install 这是安装npm的命令，npm安装好就可以使用$npm install -g hexo-cil安装Hexo

  • 然后当然要有一个编辑器啦，随便一个就行。

操作步骤

  1. Setup

  • 安装好Hexo后，自己找一个合适的位置按住Shift，鼠标右键打开Powershell，Windows是这个，别的系统不清楚

    $ hexo init <folder>  这是初始化文件夹的命令，这个folder是你自己给你网站文件取的名字
$ cd <folder>  进你创建的文件
$ npm install  安装npm

  • 用你自己的编辑器打开你建好的网站文件，找到_config.yml文件，修改配置

    title: 蒋超的博客
subtitle: '蒋超的技术博客'
description: '技术博客，编程，学习'  这是搜索引擎能搜索到的，这些关键字可以让别人容易搜索到
author: JC  这当然是作者名儿啦
url: https://jc-artisan.github.io  这个地址是用GitHub做的，你的仓库名儿是jc-artisan.github.io，jc-artisan这个是自己起的
per_page: 10  一页多少条，这个自己看着设置就行
  1. 接着在powershell里面建立自己的第一条博客
    $ Hexo new "我的第一篇博客"
    然后就去你打开的编辑器里面的source，sourse 是存放用户资源的地方，Markdown和HTML文件会被解析到public文件中，去其他文件则直接拷贝过去
    在source里面你会看到你创建的博客的md文件，这是你编写博客的地方，使用markdown语言编写，markdown是一种轻量级标记语言，很简单哦！
  • 现在就可以使用 $ Hexo g 去生成public文件
  • Hexo server 启动本地服务器 ，地址会自己出来的，这个端口号可以在_config.yml文件里面改，如果你有多个程序运行。如果没有服务器功能，安装一下服务器就好了
  1. 部署
  • 开始部署之前，必须先在 _config.yml 中修改参数，一个正确的部署配置中至少要有 type 参数
    deploy:
type: 'git'  用的git，所以部署类型就写git
repo: git@github.com:Self-madeboy/jc-artisan.github.io.git  这是你的git仓库
branch: master
  • hexo-deployer-git的安装
    $ cnpm install hexo-deployer-git --save
Hexo deploy  一键部署

    最后是我踩的坑

  • 首先就是这个cnpm的安装，这个老是出问题，我就本地全局咔咔一顿安装，然后还把cnpm的全局路径放在系统变量里面了，然后好了。
  • 然后是部署，兴冲冲的打开自己的博客，结果发现404，然后去自己建的git仓库的设置里面找到GitHub pages里面看到已经建立了，然后有个链接，点那个链接，就是你的博客
  • 我以为大功告成的时候，出来的页面没有css效果，我真是****然后发现是root没有写对，这个要写你的仓库名儿
    url: https://jc-artisan.github.io
root: /jc-artisan.github.io

    完结撒花！！！

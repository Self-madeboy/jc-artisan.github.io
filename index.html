团队协作经验

  1. 在去做下一件事情的时候，尽量理清自己手里的事情，有紧急的（比如显示bug），影响别的伙伴的，协调好再做，不然可能影响进度和浪费时间。

技术总结之线程

  1. 线程有实现接口和继承类两种实现方式，实现方法是对行为抽象
  2. 线程中的保留上下文的是上一个线程的数据，下文下就是下一个线程的准备数据

3.join可以让线程顺序执行
4.synchronized可锁同一个对象，在多个操作一个对象的时候使用，但是牺牲了性能，如果锁不同对象就会失败
5.volatile适合一写多读

  1. ThreadLocal适合用于多个对象，但牺牲空间

基础总结

  1. Java8有个字符串拼接，String.join，ArrayList有个很好用的删除元素的removeIf方法 2.mybatis-plus中mapper或者service的插入语句执行后都可以直接拿该对象继续操作。
  2. 数据库查出来的东西一定要判空，不然可能会出现空指针异常
  3. 用lambda表达式的foreach循环不能用continue，而是return，return直接开始下一轮循环。

代码规范总结

  1. 有联系的变量尽量放在一起
  2. 避免写很多if-else

3.if-else语句里面尽量不在if里面写！

需要改进的地方

  1. 对需求的了解总以为自己了解了，其实并没有，所以还是需要再细心一些。
  2. 干活比较慢，不太熟练，还需要多多练习。
  3. 代码经常写得很冗余，还是需要多思考。

mybatis通过什么方式防止sql注入?

  1. #将传入的数据都当成一个字符串，会对自动传入的数据加一个双引号。

如：where username=#{username}，如果传入的值是111, 那么解析成sql时的值为where username=”111”, 如果传入的值是id，则解析成的sql为where username=”id”.　

  1. $将传入的数据直接显示生成在sql中。

如：where username=${username}，如果传入的值是111, 那么解析成sql时的值为where username=111；
如果传入的值是; drop table user; ，则解析成的sql为：select id, username, password, role from user where username=; drop table user;

  1. #方式能够很大程度防止sql注入，$方式无法防止Sql注入。
  2. MyBatis的SQL是一个具有“输入+输出”的功能,MyBatis启用了预编译功能，在SQL执行前，会先将SQL发送给数据库进行编译；执行时，直接使用编译好的SQL，替换占位符“?”就可以了。因为SQL注入只能对编译过程起作用，所以这样的方式就很好地避免了SQL注入的问题。

什么是微服务-译文

微服务：如何更快交付软件？

随着软件组织寻找一种更快的方式去交付软件，专业术语微服务得到了很多持续关注。传统上来讲，开发团队已经构造了一个运行他们业务的单体架构，并且这些单体架构任然可以为一些依赖较少的应用工作。所以为什么这么急冲冲的去采用微服务呐？因为微服务是一个能改善软件组织速度，独立性，和创新性的引擎。

什么是微服务？

微服务是指一种不同团队用于同一目的的服务的一种架构方法，松耦合（如果每个服务必须去协同更新，他就不是松耦合）和有界上下文（如果你必须知道周围的服务，这就不是有界上下文）是这种面向服务架构的关键属性。换句话说，微服务是独立的，更小的代码片段，是一个系统的一部分但是不需要去更新和知道周围的服务。

当开发者们选择微服务和微服务设计模式去创造一个新的应用或者重构一个旧应用，这样会帮助软件跨团队和更好的进程更快的发布。微服务方法和工具让团队适应不断变化的需求，加速交付新特性给客户。

微服务模式与传统单体架构相反，传统架构由很紧密的模块组成，很少发布，不得不作为一个单元去扩展。虽然传统架构对于某些团队和某些应用还可以工作的很好，但是微服务在一些需要更敏捷和更强的拓展能力的公司很流行。

微服务的特点和操作是不同的：

  • 小型业务的集合，每个服务实现一个业务
  • 在自己的进程运行，通过HTTP API 交流
  • 能够独立于其他应用中的服务进行发布。升级，拓展和重启
  • 当被自动化系统管理时，微服务可以频繁在实时应用中频繁更新并且不会对用户造成不好的影响。

今天，各行各业–从零售到金融服务到制造业，都在使用微服务构建新的应用，分解单体架构。然而不是所有的应用都应该用微服务，因为微服务不只是单纯的代码重写，他们需要不同的思维方式，方法和操作模式。

为什么用微服务？

技术团队可以更积极的相应客户的需求。当团队采用微服务的时候可以在客户有需求的时候快速交付，不再受限于固定的发布计划。
更大的软件产出量。微服务在Agile和DevOps原则上建立，帮助团队并行工作的时候快速在离散的服务上迭代。
组织改进系统的可扩展性和可靠性。一个成功的微服务就是保持发展，他严重依赖可重复的自动化，支持细粒度的服务，使用模式设计去保持系统甚至能在单个组件故障的时候运行。

微服务 VS 传统架构

微服务

  • 聚焦一点

只做一件事， 做得很好！微服务的目标是在一个具体的的问题领域，并且包含该领域的每件事（包括数据）去拥有自己的体验，微服务的“微”是范围不是大小。

  • 是松耦合

一个为微服务架构要求服务能够自给自足，并且避免硬编码引用其他服务。

  • 是持续交付的

微服务是一个为了团队能够满足团队能够应付持续变化的需求的想法。能够尽快交付到市场，微服务可以通过自动化定期交付到产品。

  • 有独立的团队拥有服务的生命周期。

微服务的转型不仅是和团队结构有关也是和技术有关。微服务被独立的团队构造，发布，运行。不是每个服务都需要这样，但是这是一种强大的模式，对于商业型的关键服务来说。

  • 有设计模式和技术在扩展上强调分布式。

一个微服务依赖一个服务发现能力，信息，网络路由，故障检测，日志，存储，身份等等的集合。团队不能用产生一个单体架构软件相同的方法和工具去构建和运行微服务。

传统架构（单体）

  • 聚焦比较广泛

用紧密联系的软件包试图一次性解决许多业务的挑战，创造了许多代码依赖。

  • 紧耦合

系统通常是一个不谨慎按照步骤来就无法部署的相互依赖的组件缠绕的网。

  • 依赖按计划交付

应用按照计划时间被开发和更新。通常表现为按季度或者年份。

  • 有许多拥有生命周期的团队

项目团队负责构建第一代软件的迭代，然后被分开去做下一个任务。软件被交付给另一个团队去维持。

  • 有设计模式和技术，流程优先

孤立的工具和流程，专注于关键的开发阶段，QA，和发布到生产环境，产生单体软件。

当你准备去构建微服务架构的时候，要留意什么？

微服务不是任何组织或者应用都适合的。如果你不正确的应用，一个微服务架构能造成更高的耗费。如果你考虑开始作微服务，考虑一下下面的几个关键问题会有一些帮助。

  • 你的组织准备好了吗？

向微服务的转型，既是组织转型，也是技术转型。团队必须要准备接受一个自动化中心，可持续交付的方法来构建软件。你的公司准备好去移除功能上的独立和有自给自足的团队去构建和操作服务吗？你能够改变管理的进程容忍一个无人参与的部署流程吗？你考虑这些问题的方式会让你知道你准备的怎么样了。

  • 你有过于着急的开发者吗？

不是任何应用都适用微服务。匆忙去“微服务所有”，开发者大概会花费大量时间在业务不需要去改变的已有应用的编码上。使用量低的应用或者不对应服务关键性原则最好离开微服务，待在单体状态。微服务提高了敏捷性，然而带来了复杂性，确保你需要敏捷性。

  • 你的服务协调吗？

微服务之间是松耦合的，是不断变化的。你怎么找到当前服务的URL，或者流量路由到一个弹性数量的服务实例？服务之间怎么交换数据？在许多情况里面，你今天现有的技术去应对服务发现，负载均衡，消息传递对于微服务的动态变化是远远不够的。你有奉献精神和预算去投资在改变上吗？

  • 对于复杂的环境，你未来的管理是怎么样的？

随着应用和去管理的服务的数量增长，操作的风险也增长了。没有新方法去解决，在数百或者数千台服务器上创建微服务绝对会让管理人员头大。在线下的机器上打补丁或者升级是不是很困难？你能够追踪依赖然后确认在危险中的应用吗？让大量的微服务实例和最新的应用配置同步有多么困难你知道吗？你创建微服务平台的组件和这些组件和服务在哪里运行这些都会在未来的时间里对你组织的敏捷性造成重大的影响。

鉴定微服务的引导准则

你的团队写的代码比以前更多，这是微服务吗？下面是一些关于应用和系统最好去转化成微服务的

  • 成倍数的变化率

如果你的系统的一部分需要往与该系统速度和方向不一样的地方发展，就应该分开这部分转化为微服务，能让这个部分有一个独立的生命周期。（例如将电子商务系统的购物车功能从搜索或者推荐引擎中分离出来）

  • 独立的生命周期

当代码提交到一个模块的生产流程需要独立的生命周期，就把这个转化成具有自己代码库和CI/CD 管道的微服务。

  • 独立的扩展能力

当装载或者吞吐是系统独一无二的一部分，这部分也可能是拓展要求，这都让这个应用做一个微服务是一个理想的选择–以一个不同的速度拓展。

  • 孤立故障的需求

当不能选择一个故障的应用的时候，创建一个有故障转移选项的微服务去隔离不可靠的来自系统其他部分的依赖。

  • 需要去简化永久依赖的交互

当你需要在频繁变化的永久依赖中保护系统的时候，创建一个微服务（例如支付）

  • 选择的自由

通过创建微服务，允许团队去依赖他们自己优先的技术栈。

从单体到微服务的三步

抓着传统档案的团队需要去做出现代化决定。许多团队选择拆分他们的单体架构，转化为微服务，他们的最终目标是转移到原生的云应用。可用的耦合工具，最好的实践过程和技术可以让转变更加容易。

  • 步骤一：开始单体架构

我们认为单体架构是一个合适的初级阶段选择，但是当太多开发者和其他人员开始使用相同的代码库的时候，问题出现了。开发者认为产品组合很复杂，文档有限，资源太紧耦合了，他们不能学习新技能，因为他们要维护传统应用和基础设施，这就是你转去微服务的时候。

  • 步骤二：找到缝合线去发现上下文

从一个单体上去查找微服务的关键是去找到应用的“缝合口”–所以你就能知道有界上下文。然后将有界上下文取出。但是首先，你要给有界上下文放一个栅栏看看是谁在调用，然后就可以知道谁在调出，谁在调入
，依赖关系是什么，约束是什么？这会让你了解耦合，还有与之关联的事件。接着，然后你可以放一个API栅栏在有界上下文外面，然后开始拉出有界上下文到他自己的应用中。

  • 步骤三：科学的方法重构代码：Mikado 方法

对于每个改变，找到依赖关系里面出现错误的那个地方，修复他们，然后继续修复。这是一个被证明过的处理重构的深度优先的方法。

翻译 Head First Java

你能从这本书学到什么？

Head First Java 是一本完整描述Java和面向对象编程的书。这本书将教会你怎么用不一样的方法去学习Java语法和怎么去变成一个伟大的编程者。你将学会语言基础，泛型，线程，网络，分布式处理，你还会建造一个”sink the dot com”的游戏，还有网络之间的交流。

这本书有什么特殊的？

我们认为你的时间很有价值，所以使用关于认知科学方面的方式给你一个很好的学习体验，不是单纯的文字让你昏昏欲睡。

  1. 潜水

打破表面

我们会深入一点，我们会写一点代码，然后编译，然后运行。我们会谈论语法，循环，分支结构，然后看看什么让Java这么cool，你就不久就会开始编程。

Java会带你去一个新地方。从Java低调的发布1.02版本，Java用它友好的语法，面向对象的特点，存储管理，最好的一致性吸引着编程者们。一次编写随处运行强大功能的诱惑，爆发了一大批因为应对bug和体验慢如狗的速度以及那些受到平台限制的追随者。但是那是很久以前的事情了，如果你刚开始学习Java，你是幸运的。我们在前面的版本遇到了很多困难，但是你现在可以体验更快更加强大的Java。

章节一

技术的变化

如果你在软件世界呆过一段时间，你肯定会知道技术的变化很快，这是不可避免的。在我的早期职业生涯，我因为我认为我错过了技术的浪潮感到沮丧，那个时候我不喜欢技术像巴士一样，每十五分钟来一趟。
做为架构师，我们要记住我们不能停止改变，但是我们能掌管它。在接下来的章节我们会介绍过去可以教会你的，提示你不能预测未来的技术，为什么你不应该快速的丢掉传统技能。

在此前我们见过这个吗？

这些不能记住过去的最终都会重蹈覆辙
对于大多数开发者，从他们第一次学习开发语言的时候，最初的语言就成了我们的引路灯，我们通常会会根据我们最初的经历对比其他语言，然而我们最先的语言已经打下了深厚的基础，这个可能会让我们比较局限。许多开发者不怎么高兴有新特性添加到他们最爱的开发语言。当java 8 添加了lambda表达式，一些开发者不明白为什么他们的语言需要这些新特性。我相信有一部分人更钟意于过去的版本。

有经验的你可能会认识到，技术经常自我更新，想想分布式，现在很流行的微服务和无服务技术(-serverless)现在我们稍微往后退一点，看看从定制服务器虚拟机再到容器的演变。如果你斜视一点，你就能感觉到服务架构的回音。事实上，某些人可能会说微服务就是个-SOA(service-oriented archi‐
tectures)。还记得那些谈论SOA的博客，文章，书籍和会议吗？

再往后退一点，SOA是不是很像企业级的java Beans? 在不久以前，他们还只是IT技术，在一些会议上有些许的痕迹。在那个时候我们行业也继续在巨人的肩膀上继续发展，继续往后退你可能开始看到-CORBA的轮廓。

当然这不是说技术没有任何提高只是单纯的重复，云计算消除了很多制约，让我们可以共享基础设施和带来的其他便利。最重要的是去认识到今天的技术是如何被过去的技术塑造的。过去用历史技术背景塑造未来，所以你可以根据给你的历史背景做一个对未来技术发展的大致推测。

当你知道我们行业的过去，你就可以开始认识技术的行为模式了。注意到技术循环能让你跳开过去的缺陷，你还能看到为什么过去五年没有作用的技术今天能够被使用了。

但是要看清这些模式你要学习过去的技术。

学习过去的技术

温故而知新 可以为师矣 –孔子(Study the past, if you would divine the future.)

不像其他行业，我们从来不会从历史中学习，一个典型的计算机学科会要求一个编程语言的发展历程，但就是这个我们也是一笔带过。我们关注点总是最新最热的技术，热点，框架。但是没有对过去的充分接触，我们是不会领会到技术是怎么重复的。软件是一个不成熟的行业，事实上我们不喜欢谈论我们的犯的错误，但是我们应该这样，我从来不会忘记，在20078月1日，那个晚上，跨越密西西西河，35英尺高的大桥在交通高峰期倒塌，遇难13人，受伤145人。在那个礼拜的早些时候我也在那座桥。

这次事故，让全世界的职业工程师（还有工程专业的学生）都从中受教。当美国传输安全局发表了事故报道后，我保证全世界的职业工程师（工程专业的学生）都在阅读它。这样，造成这次事故的设计缺陷可能再也不会重复发生在未来的桥梁设计上。

这个错误不是我造成的

对比一下软件行业，在你最近一次参与的一个不太成功的项目的时候，你的公司文化有允许一次坦诚的会议吗，是否让你们讨论失败的原因，或者你是被迫说项目进行的很顺利？

我能明白为什么我们不会去和其他公司分享我们的失败，但是当我们不能和本公司的内部成员有一个讨论的时候，我们可能会重复犯相同的错误，还可能会重复造轮子，完全没有必要犯这些错误。

技术的旋转木马

在我的早起职业生涯，我和我的经理坐着聊天时，我问了她一个很简单的问题：为什么你决定去做经理？她说她觉得太累了去不断学习新知识，她对技术的旋转木马感到很疲惫。我承认，作为一个刚踏出学校的开发者，我不明白她的痛苦，但是经历了这么多前端框架之后，我明白了为什么她会有这种感受了。

技术一直在改变，坦率的说这是很多人被吸引进这个行业的原因。学习新事物，保持新鲜感。但是我们很多只是简历驱动实践，或者挑选技术不是为了合适，只是为了添加到我们的简历。

无价值的预测

Yogi Berra 说过：预测是一件很困难的事，尤其是预测未来的时候。这句名言可以带给你团队里面那些觉得已经了解未来技术发展空间的人。说实在的，那些你不能错过的技术的轨迹和你在这个职业生涯的时间是成比例的。

我很清楚的记得参加一个演示，是一个杰出的技术公司介绍一个吸引人的前端新技术。房间里充满了谈论的嗡嗡声。这是会议的亮点，所以人们立即就采用了新技术。在那个时候，看起来这就是技术的方向。一个我的好朋友潜心专研想抓住技术的浪潮。无数的该技术的书，博客，文章。

几年后，剧本反转，团队都快速移除这个已经在速度上抛锚的技术。用户接口不得不去重写，又开始了另一轮技术循环。即使最初的提出该技术的那家公司也很快移除了该技术。今天你几乎看不到任何开发者在net上有该技术的新代码。

纯粹的基于开发人员的兴奋，新技术看起来像是不能缺失的主题。实际上是这样，在新技术被宣布的时候，很少的人会相信我们不久会用成千上万的JavaScript构建APP。或者面向对象的C不久将会成为最受欢迎的语言，或者Clojure将Lisp带到jvm中。

我们不能预测未来，但是不会和今天相同。技术不会消亡–比如COBOL。但是我能够保证在未来五年我们都会为一个还没有被创造的东西感到兴奋。

然而我们预料不到某种具体的技术在未来会有一个主导地位，但是广泛的技术趋势却可以引导你做出决定。回顾从手动配置的服务器到虚拟机到云主机。基于架构的耗费，我们不可能回到以前对待服务器的时代。我们不能预测未来技术导向看起来会对革新的进程造成一定的扰乱。

对未来的具体某项技术做出预测是一件很蠢的事情，但是开发者去追寻最新的技术确是毫无疑问的。

过时技术的价值

使用最新的编程语言或者框架很有趣。。。但是也很令人沮丧。当你试图去安装一个0. X版本的框架，就会经历缺失依赖或者你做的结果和教程的结果不一致的痛苦。

软件行业迷恋新技术，许多开发者害怕旧的技术。没有人想去成为x开发者的最后一个。偏爱最新的技术，可能你需要领会流血一样的痛苦。但这也是一件好事当你在家做碰到你技术上的要点，但是你需要在向你的客户提出技术建议的时候更加谨慎一些。

令人兴奋的是去创造一座新桥，而不是在已经有的基础上兜兜转转。许多技术人才愿意去做开拓者，宁愿离开舒适区。没有什么比开拓新事物更有趣的事情了。

“我们的应用有四种UI技术。。。”–匿名技术专业

我打赌你会很欣赏这句话，可能一些人还会产生共鸣。我学生是怎么发现他在这种环境的？诚然，没有人有足够的力量对能够去对技术创新说不，我当然赞成团队对尝试新事物，但我们不能只是简单地追逐时尚。我们要明白新的技巧，技术，方法，框架，语言能够给我们独特的环境带来什么。

邓宁-克鲁格效应

邓宁-克鲁格效应 是一种著名的认知理论。大体上，这种理论是指能力比较好的人会低估自己的能力，然而那些经验相对较少的人会高估自己。

这种不准确的认知和在技能上的无知有关系， Donald Rumsfeld说过：“有一种知道：有些事情我们知道我们知道；有一种不知道的知道：我们知道我们不知道；但是还有我们不知道的不知道。换句话说, 能力经验少的人不知道他们不知道的东西, 当你的团队中出现一个技术迷时, 保持邓宁-克鲁格效应在心中.

过去的技术也有他的价值

我明白为什么大部分开发者更青睐新技术–雇佣性. 从而我们职业生涯的开始我们被教导对过时的技能有一种健康的恐惧, 我们不想变成我们团队里面的活化石.
我通常建议人们停留在当前的领域, 这是很重要的去记住, 我们可以用过去的技术过得不错. 用COBOL做个例子, 世界上有多少行COBOL代码, 随着会COBOL的专家越来越少, COBOL编程的日薪越来越高. 在我的职业生涯早期, 有一个快退休的同事. 他退休之后, 我们有一个退休派对. 可以想象多么令人惊讶, 我后来遇到他, 请他午餐, 他拒绝我说:”这是最黑暗的事情了, 我每周工作三天, 他们付给我的超过我全职的时候.” 我心里想着真好, 我怎么能得到那份工作呀?

我的同事会一个过时的技能, 他知道 COBOL, , JCL, TELON, IMS和其他大型机技术. 他很熟悉的旧系统, 他知道所有的骨架被埋在哪里, 因为大多数是他放置的. 他还能让这些过时的技能变成是实质的回报, 即使在退休之后.

但是COBOL只是一个例子. 多年前, 我参与一个公司合并, 其中一个我们接受的系统是用一种我们完全没有经验的技术编写的. 我们不是特例, 事实上, 像这种主要安装部分很小, 以至于国内只有少数几家承包商. 合并的时间压力没有给我们足够的时间去立即重写. 我们不得不停止合并; 除非我们能解决技术上的问题. 我们只有一个选择, 花很高的价钱请一个能够解决我们技术难题的顾问.

即便如此, 一旦我们完成了合并, 首先我们就要从我们的投资组合中移除遗留的依赖和昂贵的承包商. 同样的, 最符合你利益的是掌握新技能. 有时候我们太快的把过去的技术从我们的简历里面拿掉, 有时候我们过于停留在旧技术, 挑战就是知道什么时候从旧技术到新技术.
